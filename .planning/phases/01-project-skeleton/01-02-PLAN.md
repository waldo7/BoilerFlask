  </read_first>
  <action>
    Overwrite `app/__init__.py` (created empty in Task 1). Per D-01 architecture rule (AGENTS.md): use app factory pattern with `create_app()`.

    **Module-level imports:** Only import `os`, `Flask` and `render_template` from flask. No project-internal imports at module level.

    **create_app(config_name=None):**
    - Determine config_name: if None, read `os.environ.get('FLASK_ENV', 'development')`
    - Map config_name to config class: 'production' → ProductionConfig, 'testing' → TestingConfig, else → DevelopmentConfig. Import the target class INSIDE the if-branch.
    - Create `app = Flask(__name__)`, call `app.config.from_object(ConfigClass)`
    - Import main blueprint from `app.main` INSIDE the function: `from app.main import main_bp` then `app.register_blueprint(main_bp)`. Per PITFALLS.md §1: blueprint import goes inside the factory function, not at module level.
    - Call `register_error_handlers(app)` — defined as a separate function in this file.
    - Return `app`.

    **register_error_handlers(app):**
    - Define a dictionary `error_pages` mapping HTTP codes to data dicts with keys: `title`, `message`, `icon` (per D-11, D-12). Exact values per RESEARCH.md Component Specification:
      - 403: title="Access Forbidden", message="Sorry, you don't have permission to access this page.", icon="bi-shield-lock"
      - 404: title="Page Not Found", message="Oops! The page you're looking for doesn't exist.", icon="bi-question-circle"
      - 500: title="Internal Server Error", message="Something went wrong on our end. Please try again later.", icon="bi-exclamation-triangle"
    - Define an inner closure `make_handler(code, title, message, icon)` that returns a handler function. The inner handler calls `render_template('error.html', code=code, title=title, message=message, icon=icon)` and returns the tuple `(rendered_response, code)` — Flask error handlers must return the status code.
    - For each (code, data) in error_pages: call `app.register_error_handler(code, make_handler(code, data['title'], data['message'], data['icon']))`. Use `register_error_handler()` — NOT the decorator pattern — because decorators don't work inside functions (per RESEARCH.md verification).
    - Per D-12: handlers use a single shared `error.html` template.

    The `app/auth` blueprint (forward-compatible placeholder) is NOT registered in Phase 1.
  </action>
  <verify>

  </verify>
