- GET /auth/register returns 200 with 'Create Account' in response
- POST /auth/register with valid data creates user, redirects to /dashboard (302 or 200 after follow), auto-login so /dashboard shows welcome
- POST /auth/register with existing email shows 'Email already registered.' (verify form re-rendered with error)
- POST /auth/register with weak password (< 8 chars) shows 'Password must be at least 8 characters.'
- POST /auth/register with password lacking 2 character types shows complexity error

**AUTH-02: Login + Session (6 tests)**
- GET /auth/login returns 200 with 'Log In' in response
- POST /auth/login with valid credentials redirects to /dashboard (302), subsequent GET /dashboard returns 200 (not redirected — session active)
- POST /auth/login with 'remember me' checked — verify session cookie has longer expiry (check response headers for Set-Cookie expiration)
- POST /auth/login with bad password returns 200 (form re-rendered) with 'Invalid email or password.' flash
- POST /auth/login with deactivated account (is_active=False) is rejected with 'This account has been deactivated.'
- Already-logged-in user visiting GET /auth/login redirects to /dashboard with 'You are already logged in.' flash

**AUTH-03: Logout (2 tests)**
- GET /auth/logout returns 302 redirect to /
- After logout, accessing /dashboard redirects to /auth/login (session cleared)

**AUTH-04: Password Reset (6 tests)**
- GET /auth/forgot-password returns 200 with 'Forgot Password' heading
- POST /auth/forgot-password with existing email renders 'Check Your Email' page (D-06: generic response)
- POST /auth/forgot-password with non-existent email renders same 'Check Your Email' page (D-06: prevents enumeration)
- GET /auth/reset-password/<valid_token> returns 200 with 'Set New Password' heading
- GET /auth/reset-password/<invalid_token> flashes error and redirects to /auth/forgot-password
- POST /auth/reset-password/<valid_token> with new password redirects to /auth/login, then user can login with new password
- Token reuse: POST same token twice — second attempt fails because token is one-time use (D-07)

**AUTH-05: Roles (3 tests)**

Key implementation notes for specific tests:

**test_register_success:** POST /auth/register with valid data, follow_redirects=True. Assert final response has b'Welcome' or b'Dashboard'. Also query User directly: user = User.query.filter_by(email='new@example.com').first(); assert user is not None; assert user.role == Role.USER; assert user.check_password('Test1234!').

**test_register_duplicate:** First create user via fixture or direct insert. POST register with same email. Assert response status 200 (form re-rendered), assert b'Email already registered.' in response.data.

**test_login_session_persists:** POST /auth/login, follow_redirects=False. Assert 302 redirect. Then GET /dashboard with SAME client (session cookie preserved). Assert 200 with b'Welcome'.

**test_login_remember_me:** POST with remember=True. Check response Set-Cookie header for expires/max-age. The session cookie should have a future expiration. Approximate check: assert the cookie string contains 'Expires' or 'Max-Age'.

**test_login_deactivated_user:** Create user via `_db` session, set `user.is_active = False`, `db.session.commit()`. POST login. Assert response has b'This account has been deactivated.'.

**test_logout_clears_session:** Use auth_client. POST /auth/logout (or GET since logout is GET). Assert redirect to /. Then GET /dashboard — assert redirected to /auth/login.

**test_forgot_password_existing_email:** POST /auth/forgot-password with user's email. Assert response has b'Check Your Email'.

**test_forgot_password_unknown_email:** POST with 'nonexistent@example.com'. Assert response ALSO has b'Check Your Email' (D-06 enumeration prevention).

**test_reset_password_valid_token:** Generate token with `token = generate_reset_token(user)`. GET /auth/reset-password/{token}. Assert 200, b'Set New Password'. POST with new password 'NewTest1234!'. Assert redirect to /auth/login (302). Then verify new password works: POST /auth/login with 'NewTest1234!' succeeds.

**test_reset_password_invalid_token:** GET /auth/reset-password/bad-token. Assert 302 redirect to /auth/forgot-password (or follow redirect and check flash).

**test_reset_password_token_one_time_use (D-07):** Generate token. POST reset successfully. Try same POST again — should fail (302 redirect or form re-rendered with error).

**test_default_role_on_register:** After successful POST /auth/register, query User by email and assert user.role == Role.USER.

**test_role_enum_values:** assert Role.USER.value == 'user'; assert Role.ADMIN.value == 'admin'; assert Role.SUPERUSER.value == 'superuser'.

**test_dashboard_redirects_unauthenticated:** client.get('/dashboard', follow_redirects=False). Assert 302. Assert '/auth/login' in response.headers['Location'].

**test_dashboard_authenticated:** auth_client.get('/dashboard'). Assert 200. Assert b'Welcome, test@example.com' in response.data. Assert b'Getting Started' in response.data.

**test_email_normalization:** POST register with 'Test@Example.com  ' (whitespace + mixed case). POST login with 'test@example.com' (lowercase, no whitespace). Assert login succeeds (302 redirect).

**test_login_already_authenticated:** Use auth_client to GET /auth/login. Assert 302 redirect to /dashboard. Follow redirect, check for b'You are already logged in.' (info flash).

**test_csrf_config:** Assert app.config.get('WTF_CSRF_ENABLED') == False in TestingConfig (tests run with CSRF disabled for simplicity, but config tracks the setting).

Each test should be self-contained and cleanup after itself (the `_db` fixture handles table creation/drop). Use `_db` fixture when tests need direct DB access.
